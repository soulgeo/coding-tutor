\chapter{Pedagogical Design}

\section{Educational Objectives and Target Audience}
The main goal of the "Coding Tutor" application is to provide a clear and solid introduction to Python programming. The curriculum is designed specifically for beginners, using a step-by-step approach where each unit builds directly on the logic learned in previous lessons. 

\section{Curriculum Structure and Organization}
The course content is divided into six units, defined in the system's configuration file (\texttt{units.json}). This structure gradually increases in difficulty, guiding the user from basic syntax to more complex concepts:

\begin{itemize}
    \item \textbf{Unit 1: Python Foundations} – Introduction to variables and printing.
    \item \textbf{Unit 2: Decision Making} – Logical operators and conditional statements (if/else).
    \item \textbf{Unit 3: Loops \& Automation} – Repeating actions and the \texttt{range()} function.
    \item \textbf{Unit 4: Data Collections} – Managing grouped data using lists.
    \item \textbf{Unit 5: Reusable Tools} – Organizing code with custom functions.
    \item \textbf{Unit 6: Building Projects} – Combining learned skills through practical exercises.
\end{itemize}

\section{Self-Assessment and Active Participation}
Unlike passive video tutorials, the app keeps students engaged with interactive "missions." Each lesson includes a coding challenge where the student's code is checked against the correct answer (\texttt{expectedOutput}) using the Piston API. This provides the instant feedback that is so important for self-paced learning.

\section{Adaptive Learning and Individualization}
To personalize the learning experience, the application includes a smart hint system that adapts to how well the student is doing.

\subsection{The Hint Trigger Mechanism}
The system tracks how the user's code performs. If a student's output doesn't match the \texttt{expectedOutput}, the app counts this as a failed attempt ($f$). The hint system follows this simple logic:

\[
H(f) = 
\begin{cases} 
\text{Hidden} & \text{if } f < 2 \\
\text{Visible} & \text{if } f \geq 2 
\end{cases}
\]

Waiting until the second failed attempt encourages students to try debugging their own code first. This promotes a healthy "trial-and-error" learning style before stepping in to help.


\chapter{Technical Implementation}

\section{System Architecture Overview}
The application is built using a modern, separated architecture with three main parts:
\begin{itemize}
\item \textbf{Frontend Layer:} A React application written in TypeScript, using Tailwind CSS to look great on any screen size.
\item \textbf{Execution Layer:} A Dockerized Piston engine that safely runs the students' Python code in an isolated environment.
\item \textbf{Persistence Layer:} Google Firebase for saving progress in real-time and securely managing user logins.
\end{itemize}

\section{Content Management and Data Pipeline}
One of the best technical features of the platform is how easily new lessons can be added, keeping the course content completely separate from the app's code.

\subsection{The Synchronization Engine}
Lessons are written as standard Markdown files with a YAML header. However, the React app doesn't load these files directly. Instead, a Node.js script (\texttt{sync\_lessons.js}) processes them, grabs the important data (like \texttt{pretypedCode} and \texttt{expectedOutput}), and uploads it straight to the Firebase database. This keeps the web app lightweight and fast.

\subsection{Data Fetching and Rendering}
When a user opens a lesson, the \texttt{LessonPage} component asks Firebase for the data using the lesson's \texttt{id}. Once the text is downloaded, a custom \texttt{MarkdownRenderer} turns it into the interactive interface the student sees. 

To make reading easier for beginners, this renderer uses CSS to turn off "font ligatures." This means that symbols like \texttt{<=} or \texttt{!=} stay as two separate characters (just like they look on a keyboard) rather than merging into a single complex symbol.

\section{Sandboxed Execution Engine}
Running code submitted by users can be dangerous. To keep the server safe, the platform uses Piston, a fast and secure code execution engine running inside Docker.

\subsection{Container Isolation}
The \texttt{docker-compose.yml} file sets up the Piston service. Piston uses \texttt{nsjail} to create temporary, locked-down environments for every piece of code. This prevents the code from accessing the main network or reading private files. Additionally, a startup script ensures Python 3.12.0 is installed the moment the server turns on, so students never have to wait for their code to run.

\section{Assessment Logic and Telemetry}
The \texttt{LessonPage.tsx} component handles the main learning loop through two different actions: Trial and Validation.

\subsection{Trial vs. Validation Workflows}
\begin{itemize}
    \item \textbf{Trial (\texttt{runCode}):} The user's code is sent to Piston, and the result (\texttt{stdout} or \texttt{stderr}) is printed on the screen. This lets students experiment and test things safely without recording any "failed" attempts in the database.
    \item \textbf{Validation (\texttt{submitCode}):} This is the actual grading system. The result is cleaned up using \texttt{.trim()} and strictly compared against the database's \texttt{expectedOutput}. 
\end{itemize}

\subsection{Data Persistence and Progress Tracking}
When a student successfully submits a lesson, the system updates their profile in Firebase. It adds the lesson's ID to their \texttt{completedLessons} list. This reliable tracking is what powers the real-time statistics shown on the student's Dashboard.

\section{Database Schema}
To provide rich and helpful statistics, the system uses a flexible NoSQL structure in Firebase Firestore.

\subsection{User Data Model}
Each user profile (\texttt{UserData}) keeps a detailed log of their learning journey:
\begin{itemize}
\item \textbf{Completed Lessons:} A list of the lesson IDs the student has successfully finished.
\item \textbf{Attempt Tracking:} A counter that remembers how many times a user tried to solve a specific lesson before passing.
\end{itemize}

\section{Security and Authentication}
User logins are handled securely by Firebase Auth. The app uses an \texttt{AuthContext} provider to ensure that only logged-in students can view the lessons or their personal dashboards.

\chapter{User Interface and Experience Design}
\label{ch:ui_ux_design}

\section{Design Philosophy}
The visual design of the "Coding Tutor" application is rooted in minimalism and cognitive load reduction. For a beginner, learning to program is already a complex mental task. Therefore, the interface was designed to be as unobtrusive as possible, removing unnecessary visual clutter and utilizing a clean, high-contrast aesthetic that keeps the user's focus entirely on the educational content.

\section{Dashboard and Information Architecture}
Upon logging in, the student is presented with the Dashboard. The primary user experience (UX) goal of this screen is to provide immediate, encouraging feedback on the student's overall progress.

\subsection{Visualizing Progression}
The curriculum is broken down into a grid of distinct "Unit" cards. Rather than relying solely on text or simple checklists, the interface utilizes bold, circular progress indicators for each unit. This visual approach allows the student to instantly grasp their standing in the course at a glance. Partially filled rings act as a subtle psychological motivator, encouraging the student to complete the remaining lessons to "close the circle."

\subsection{Performance Metrics}
At the top of the dashboard, a prominent statistics banner displays the user's total code submissions and average attempts per lesson. Presenting these metrics in large, clear typography transforms standard database tracking into a gamified element. It shifts the user's goal from simply completing lessons to improving their coding accuracy over time.

\section{The Dual-Pane Learning Environment}
The most critical screen in the application is the Lesson Page. The primary design challenge here was presenting both theoretical instruction and an interactive coding workspace without overwhelming the user.

To solve this, the desktop interface employs a strict dual-pane, side-by-side layout. 
\begin{itemize}
    \item \textbf{The Theory Pane (Left):} This area is dedicated to reading. It uses a light, clean background with generous margins and distinct headings to make the markdown content easily scannable.
    \item \textbf{The Interactive Workspace (Right):} This area houses the code editor and console. 
\end{itemize}

This side-by-side design is a deliberate pedagogical choice. It eliminates the need for the student to constantly scroll up and down between the instructions and their code, preventing the loss of context and maintaining their flow state.

\section{The Interactive Workspace}
The right side of the Lesson Page was designed to look and feel like a professional programming environment, while remaining approachable for absolute beginners.

\subsection{Editor and Console Distinction}
The code editor features a clean, light interface with prominent line numbers to help students track their logic. Directly below the editor is the output console. The console deliberately uses a dark background and monospace font. This high-contrast visual separation clearly distinguishes the code the student writes from the response the computer gives, mimicking the look of a real developer terminal.

\subsection{Action Buttons and Visual Weight}
The interactive actions are governed by two distinct buttons, designed with clear visual hierarchy:
\begin{itemize}
    \item \textbf{Run Code (Secondary Action):} Styled with a muted, outlined appearance. This signals that clicking it is a safe, low-stakes action meant for testing and debugging.
    \item \textbf{Submit (Primary Action):} Styled with a solid, bold color block. This high visual weight draws the user's eye and clearly indicates the final, formal action of having their code graded.
\end{itemize}

\subsection{Post-Success Interface Updates}
A crucial aspect of the UX is how the interface adapts once a student successfully completes a lesson:
\begin{itemize}
    \item \textbf{Subtle State Shifts:} The "Submit" button gracefully shifts its thematic color (from the accent color to the primary color) to indicate the completed state.
    \item \textbf{Navigation Unlock:} The completed state is passed to the sticky top navigation bar, updating the unit's progress bar and unlocking the path to the next lesson.
\end{itemize}

\section{Typography for Code Legibility}
Reading code requires different typographic considerations than reading standard text. The interface ensures that the student can easily spot small syntax errors.

To aid in the cognitive parsing of raw code, the interactive editor integrates robust syntax highlighting tailored specifically for Python. By assigning distinct, high-contrast colors to different semantic elements—such as keywords, strings, variables, and comments—the interface allows beginners to visually dissect the structure of their logic. This immediate visual feedback helps students quickly identify missing quotation marks or unclosed parentheses before even attempting to run their code.

\section{Non-Intrusive Feedback Mechanisms}
In educational software, the way the system communicates success or failure drastically impacts the user's emotional state. The application explicitly avoids native browser alerts, which are jarring and disruptive. Instead, it utilizes a custom "Toast" notification system that slides smoothly onto the screen.

\subsection{Pedagogical and Success Feedback}
During the lessons, these toasts provide instant feedback. When a user submits an incorrect answer, a gentle error toast encourages them to try again. When they succeed, the system displays a highly visible success toast containing playful, context-aware messages (e.g., "You took the \texttt{else} path to victory!"). This specialized feedback rewards the student while reinforcing the programming concepts they just learned.

\subsection{Authentication Error Handling}
The same elegant feedback system is applied to the login and registration flows. Instead of exposing raw database errors (such as \texttt{auth/invalid-credential}), the application intercepts these codes and translates them into friendly, human-readable toast notifications (e.g., "The email or password you entered is incorrect"). This ensures that even at the very first touchpoint of the application, the user feels guided and supported.
